# 第6章 データベースとパフォーマンス ～インデックス設計～ 学習ノート

## 演習作成の意図

第6章のテーマはインデックス設計。書籍ではB-treeの仕組みやインデックスが効くケース・効かないケースが解説されるが、パターンを暗記するだけでは実務で応用が利かない。そこで、具体的なSQL文を提示して「どの列にインデックスを貼るか」「なぜ効く/効かないか」をB-treeの構造から説明する形式にした。単一インデックスから複合インデックス、さらに「貼るべきでないケース」まで段階的に難度を上げ、第5章で学んだ非正規化との使い分けに接続させることが狙い。

## 演習の題材

ECサイトの商品検索・注文管理システム（customers, products, orders, order_items の4テーブル）。
データ規模: products 10万件、orders 1,000万件、order_items 3,000万件、customers 100万人を想定。

---

## Q1: インデックス設計の基礎

### Q1(a): カテゴリ絞り込み
```sql
SELECT * FROM products WHERE category = '家電' AND is_published = TRUE;
```
- **回答**: categoryにインデックスを貼る
- is_publishedは2値しかないのでカーディナリティが低く効果が薄い

### つまずいたポイント: インデックスはWHERE句の列に貼る
- 最初にproduct_nameと答えてしまった。カーディナリティが高い列を選ぶ考え方は正しかったが、**インデックスはWHERE句で検索条件に使われている列に貼る**という基本を見落としていた
- SQLを書く機会が減っていることが原因かもしれない。書籍を読むだけでは流してしまう基本的な部分

### Q1(b): 顧客の注文履歴
```sql
SELECT * FROM orders WHERE customer_id = 123 ORDER BY order_date DESC;
```
- **回答**: customer_idにインデックスを貼る
- 100万人分あるのでカーディナリティが高い
- (customer_id, order_date)の複合インデックスにすれば絞り込み+ソート両方をインデックスで処理できる

### Q1(c): 在庫切れ商品の抽出
```sql
SELECT * FROM products WHERE stock_qty <= 5 AND is_published = TRUE;
```
- **回答**: stock_qtyにインデックスを貼る
- 最初「比較演算子があるとインデックスが効かない」と誤解していた

### つまずいたポイント: 比較演算子とインデックス
- `<=` などの比較演算子はB-treeインデックスが**効く**
- B-treeはデータがソートされた状態なので範囲検索に対応できる
- **効かない**のは列を関数等で**加工**するケース（例: `WHERE UPPER(col) = ...`）
- 「効く・効かない」を言葉で覚えるのではなく、B-treeの構造から**なぜそうなるか**を理解する必要がある

#### ※補足: なぜ比較演算子でインデックスが効くのか（B-treeの処理の観点）

B-treeは値をソート済みの状態で保持している。例えばstock_qtyのインデックスは以下のような構造:

```
          [15]
        /      \
    [3, 5]    [30, 50, 80, 150, 200]
```

`WHERE stock_qty <= 5` を実行すると:
1. ルートノードの[15]と5を比較 → 5は15より小さいので左の子ノードへ
2. 左の子ノード[3, 5]に到達 → 5以下の値を左から順に取得
3. **5を超えた時点で探索終了**

ソート済みだからこそ、「5以下」という条件に合致する範囲の**開始位置と終了位置**がB-treeを辿るだけで特定でき、それ以外のノードは一切見る必要がない。

一方、`WHERE UPPER(product_name) = ...`のように列を加工すると、インデックスに格納されている元の値とは**別の値**で比較することになるので、ツリーを辿って位置を特定できない。

まとめ:
- **比較演算子**: インデックスのソート順をそのまま利用して範囲を特定できる → **効く**
- **列の加工**: インデックスのソート順と無関係な値になる → **効かない**

## Q2: インデックスが効かないケース

### Q2(a): 列の関数加工
```sql
-- NG: インデックスが効かない
SELECT * FROM products WHERE UPPER(product_name) = 'USB-C ハブ 7IN1';
-- OK: 列を加工せずそのまま比較
SELECT * FROM products WHERE product_name = 'USB-C ハブ 7in1';
```
- インデックスは元の値に対して構築されているので、関数で加工された値は探せない

### Q2(b): ORの使用
```sql
-- NG: ORがあるとインデックスが効きにくい
SELECT * FROM orders WHERE order_date BETWEEN '2024-01-01' AND '2024-03-31'
    OR status = 'pending';
-- OK: UNIONに分解
SELECT * FROM orders WHERE order_date BETWEEN '2024-01-01' AND '2024-03-31'
UNION
SELECT * FROM orders WHERE status = 'pending';
```
- ※ORが効かないケースは書籍には明示的に記載されていなかった

### 振り返り: ORでインデックスが効きにくい理由
ORは「どちらかの条件を満たす行」を返す。B-treeインデックスは1つの条件に対して効率的にツリーを辿れるが、ORで2つの異なる列の条件が並ぶと、1つのインデックスで両方の条件をカバーできない。結果的にオプティマイザはフルスキャンを選択することがある。UNIONに分解すれば、各クエリが独立してそれぞれのインデックスを使える。

### Q2(c): 中間一致LIKE
```sql
-- NG: 中間一致はインデックスが効かない
SELECT * FROM products WHERE product_name LIKE '%イヤホン%';
```
- B-treeはデータを辞書順にソートして保持している
- 前方一致（`'ワイヤレス%'`）なら「ワ」の位置から辿れる
- 中間一致（`'%イヤホン%'`）だと開始位置が分からず全ページめくるしかない＝フルスキャン

### Q2(d): 列の関数加工（日付）
```sql
-- NG: TO_CHARで列を加工している
SELECT * FROM orders WHERE TO_CHAR(order_date, 'YYYY-MM') = '2024-03';
-- OK: 日付の範囲検索にする
SELECT * FROM orders WHERE order_date >= '2024-03-01' AND order_date < '2024-04-01';
```
- Q2(a)と同じパターン。暗黙の型変換ではなく、列を関数で加工しているケース

## Q3: 複合インデックスの設計

### Q3(a): 列の順序
- **回答**: (category, price, product_name)の順
- 等値検索（category =）を先に、範囲検索（price BETWEEN）を後に置く
- 等値検索で先にピンポイントに絞り込み、その結果に対して範囲検索をかける方が効率的

### Q3(b): 複合インデックスが効くクエリ・効かないクエリ
1. `WHERE category = '家電' AND price BETWEEN 1000 AND 5000;` → **効く**（左からcategory, priceと順に使える）
2. `WHERE price BETWEEN 1000 AND 5000;` → **効かない**（左端のcategoryを飛ばしている）
3. `WHERE category = '家電' AND product_name LIKE 'ワイヤレス%';` → **categoryだけ効く**（priceを飛ばしてproduct_nameは効かない）

### つまずいたポイント: 複合インデックスのB-tree構造
- 最初、各クエリにそれぞれ個別のインデックスが貼られていると誤解していた
- 複合インデックスは**1つのB-tree**であり、ノードに格納されるキーが**タプル（組）**になる

```
単一インデックス: [家電] [スポーツ] [書籍]
複合インデックス: [家電, 4500] [家電, 15000] [スポーツ, 2500] ...
```

- 比較は左の列から順に行われるため、**左端の列がないとツリーの最初の分岐点でどちらに進むべきか判断できない**
- B-treeの分割・昇格の仕組みは単一インデックスと全く同じ。違いはキーがタプルになるだけ

## Q4: インデックスを貼るべきでないケース

### Q4(a): 低カーディナリティ（is_published: TRUE/FALSE）
- 10万件中90%がTRUEなら9万件ヒット。インデックスで9万件を辿るよりフルスキャンの方が速い
- インデックスが効果的なのは検索結果が全体のごく一部に絞られる場合

### Q4(b): 全件取得（価格順に全商品一覧）
- 全件取得する時点でインデックスを経由する意味がない

### Q4(c): 低頻度クエリ（月1回のバッチ処理）
- インデックスは維持コストがかかる（INSERT/UPDATE/DELETEのたびにB-treeも更新）
- 月1回の恩恵のために日常の書き込み性能を犠牲にするのはコストに見合わない

---

## Q5: 第5章との接続（非正規化 vs インデックス）

| 状況 | 対応 |
|---|---|
| WHERE句にカーディナリティの高い列がある | **インデックス**で解決 |
| 全件集約（COUNT, AVG, SUM等）が重い | **非正規化**（サマリ列）を検討 |
| 実行頻度が低い | **どちらも不要**の可能性 |

- インデックスを貼るべきでないケース（Q4）に該当し、かつパフォーマンス改善が必要な場合に非正規化を検討する
- ただしバッチ処理のようにリターンが小さいケースは非正規化もインデックスも不要な可能性がある
- WHERE句にカーディナリティの高い列がなく、大規模テーブルの結合・集約が避けられない場合が非正規化の出番

## 振り返り: 本質的に理解できていなかったこと

この演習を通して、以下の点が「知識として知っていても本質的に理解できていなかった」ことに気づいた:

1. **インデックスはWHERE句の列に貼る**という基本。SQLを書く機会が減っていたため意識できていなかった
2. **比較演算子(<=, >=等)はインデックスが効く**のに「効かない」と誤解していた。B-treeがソート済みデータを保持する構造だから範囲検索に対応できるという原理を理解していなかった
3. **インデックスが効く・効かないパターン**について、パターンとして暗記するのではなく、B-treeの内部構造からなぜそうなるかを理解する必要がある
4. **複合インデックスのB-tree構造**。キーがタプルになり左から順に比較されるという仕組みを理解したことで、「左端の列から使う必要がある」理由が腑に落ちた

書籍を読むだけでは流してしまう部分が、演習で実際に考えることで気づけた。特にB-treeの内部処理を具体的に考えることの重要性を実感した。
