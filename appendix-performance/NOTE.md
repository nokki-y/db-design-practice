# 付録: 正規化 vs インデックス vs 非正規化 パフォーマンス比較

## 演習作成の意図

第5章・第6章で「全件集約にはインデックスが効かない → 非正規化の出番」「インデックスは選択率が高いときに効く」と理論的に学んだが、実際の数字を見ていないと「どの程度効くのか」の感覚が掴めない。そこで、大量データ（orders 100万件、order_items 300万件）を用意し、選択率の異なる4パターンのクエリで「ベースライン → インデックス追加 → 非正規化」の実測比較を行う形式にした。理論を数字で裏付け、判断の解像度を上げることが狙い。

## 概要

正規化されたテーブルに大量データを投入し、以下の3パターンでパフォーマンスを比較する。

1. **ベースライン**: 正規化テーブル（インデックスなし ※PKの自動インデックスのみ）
2. **インデックス追加**: 正規化テーブル + 外部キーにインデックス
3. **非正規化（集計サマリーテーブル）**: 集計結果を事前に持つテーブル

## テーブルサイズ

| テーブル | 件数 | サイズ |
|----------|------|--------|
| perf_categories | 20 | 24 kB |
| perf_customers | 10,000 | 1,032 kB |
| perf_products | 1,000 | 144 kB |
| perf_orders | 1,000,000 | 77 MB |
| perf_order_items | 3,000,000 | 276 MB |
| perf_monthly_category_sales（集計サマリー） | 720 | 極小 |

## 各クエリの特徴

### Q1: 月別・カテゴリ別の売上集計（全件集計）

```
WHERE句: なし（全件集計）
JOIN: 4テーブル（order_items → orders → products → categories）
集計対象: 300万件の order_items 全行
```

**ポイント**: WHERE句がないため、300万件を全件読む必要がある。インデックスは「大量の行から少数を絞り込む」ときに効くものなので、全件集計ではオプティマイザがSeq Scan（順次スキャン）を選ぶ。インデックスがあっても使われない。

**効果的な非正規化**: 集計結果を事前にサマリーテーブルとして持つことで、300万件 → 720行に削減。読む行数が激減するため劇的に高速化される。

### Q2: 特定顧客の注文履歴（高選択性の検索）

```
WHERE句: customer_id = 42 AND order_date >= '2025-01-01'
選択性: 非常に高い（100万注文のうち約30件 = 0.003%）
JOIN: 3テーブル（orders → order_items → products）
```

**ポイント**: `customer_id = 42` で100万注文 → 約100件に絞り込み、さらに `order_date` で約30件に。インデックスがあれば、この30件をピンポイントで取得でき、残り99万9,970件は一切読まない。**インデックスが最も効くパターン**。

### Q3: 特定期間の売上集計（範囲検索）

```
WHERE句: order_date BETWEEN '2025-04-01' AND '2025-06-30'
選択性: 中程度（3年分のうち3ヶ月分 ≒ 約8%、注文約8.3万件・明細約25万件）
JOIN: 4テーブル（orders → order_items → products → categories）
```

**ポイント**: Q2と違い、WHERE句はあるが選択率が約8%と中程度。インデックスは使われるが、対象行が多いため劇的な高速化にはならない。Q2（0.003%）との選択率の違いがインデックスの効き方に直結する。

### Q4: 特定都道府県の顧客別売上（低カーディナリティの検索）

```
WHERE句: prefecture = '東京都'
選択性: 低い（47都道府県の1つ ≒ 約2%の顧客だが、その注文・明細は約7万件）
カーディナリティ: 47（値の種類が少ない）
JOIN: 3テーブル（customers → orders → order_items）
```

**ポイント**: `prefecture` のカーディナリティは47しかない。インデックスで顧客を231人に絞れるが、その先のorders・order_itemsの結合で大量の行を処理する必要がある。customers側のインデックスは効くが、全体の処理時間への影響は限定的。Q2（customer_id: カーディナリティ1万）と比べて、カーディナリティの違いがインデックスの効果にどう影響するかが見どころ。

## 計測結果

| クエリ | ベースライン | +インデックス | 非正規化（集計サマリー） | 備考 |
|--------|---:|---:|---:|---|
| Q1: 全件集計 | ~5,100 ms | ~4,900 ms | **1.6 ms** | インデックスあり/なしで差なし（誤差範囲）。サマリーで3,000倍高速 |
| Q2: 高選択性（0.003%） | 142 ms | **2 ms** | - | インデックスで**70倍高速** |
| Q3: 範囲検索（8%） | 539 ms | **403 ms** | - | インデックスで約25%改善。効くが劇的ではない |
| Q4: 低カーディナリティ（2%） | 177 ms | **166 ms** | - | インデックスの効果はわずか（約6%改善） |

※ Q1は複数回計測して安定値を確認。初回計測では「インデックスありの方が遅い」と見えたが、複数回実行すると誤差範囲だった。オプティマイザは正しくインデックスを無視し、両方ともSeq Scanを選択している。つまりQ1のような全件集計では**インデックスはSELECTの速度に影響しない**。

## 考察

### 第5章・第6章の理論が数字で裏付けられた

第5章で「全件集約にはインデックスが効かない → 非正規化の出番」、第6章で「インデックスはWHERE句で選択率が高いときに効く」と学んだが、当時は理論として理解しただけだった。今回、実際に数字を見ることで腹落ちした。

Q1（全件集計）は複数回計測した結果、インデックスの有無でSELECTの実行時間に差はなかった（初回計測の差は誤差）。オプティマイザが正しくインデックスを無視してSeq Scanを選んでいるため。ただし「SELECTが遅くならないなら損はない」かと言うとそうではない — コストは書き込み側にかかっている（後述の「インデックスの本当のコスト」参照）。

### 選択率がインデックスの効果を決める — Q2 vs Q3 vs Q4の比較

| クエリ | 選択率 | インデックスの効果 |
|--------|--------|-------------------|
| Q2: customer_id = 42 | 0.003% | 142ms → 2ms（**70倍**） |
| Q3: order_date 3ヶ月分 | 8% | 539ms → 403ms（25%改善） |
| Q4: prefecture = '東京都' | 2% | 177ms → 166ms（6%改善） |

第6章で「カーディナリティが高い列にインデックスを貼る」と学んだが、この結果を見ると**カーディナリティだけでなく、実際にヒットする行数（選択率）がインデックスの効きを決める**ことが分かる。

Q4はprefecture = '東京都'で顧客を231人に絞り込めているが、その231人の注文（約2.3万件）× 明細（約7万件）を結合する処理が依然として重い。つまり**インデックスで入口（customers）は絞れても、その先のJOINで扱う行数が多ければ全体の改善は限定的**。第6章のQ1(a)で「is_publishedのような2値の列にインデックスを貼っても、9万件ヒットするなら意味がない」と学んだが、それと本質的に同じ構造。

Q3は選択率8%と中程度で、EXPLAIN ANALYZEを見るとBitmap Index Scanが使われている。Seq Scanではないのでインデックスは**使われている**が、8万件以上を読んでいるため劇的な改善にはならない。第6章で学んだ「B-treeで範囲の開始位置と終了位置を特定できる」仕組みがここで活きている。

### 第5章Q5の判断フレームワークを改めて検証する

第6章の最後にまとめた判断基準:

| 状況 | 対応 |
|---|---|
| WHERE句にカーディナリティの高い列がある | **インデックス**で解決 |
| 全件集約（COUNT, AVG, SUM等）が重い | **非正規化**（サマリ列）を検討 |

今回の計測結果はこの判断基準と完全に一致する。Q2はインデックスだけで70倍高速化でき、Q1は非正規化（サマリーテーブル）で3,000倍高速化できた。

ただし今回のQ3・Q4で見えたのは、**この2択に当てはまらないグレーゾーンがある**ということ。「WHERE句はあるがカーディナリティが低い/選択率が高い」ケースでは、インデックスの効果は限定的だが、全件集約ほど重くもない。このグレーゾーンでの実務判断としては:
- そもそも許容できるレスポンスタイムかどうかで判断する（Q4の166msは多くの場面で十分速い）
- 許容できないなら、パーティショニングやマテリアライズドビュー等の別の手段を検討する

### インデックスの本当のコスト —「貼っておけば損はない」は誤り

Q1のSELECTではインデックスの有無で速度差がなかった。オプティマイザが賢く無視してくれるなら「貼っておいて損はない」のでは？ — この疑問に答えるために、perf_order_items への10万件INSERTの速度を計測した。

**perf_order_itemsへの10万件INSERT（3回計測の結果）:**

| 状態 | 1回目 | 2回目 | 3回目 | 平均 |
|------|---:|---:|---:|---:|
| インデックスなし（PKのみ） | 911 ms | 845 ms | 886 ms | **881 ms** |
| インデックスあり（PK + order_id + product_id） | 1,549 ms | 1,386 ms | 1,269 ms | **1,401 ms** |

→ インデックスが2つ増えるだけで、INSERTが**約60%遅くなった**。

#### なぜ遅くなるのか

1行INSERTするごとに、テーブル本体への書き込みに加えて**すべてのインデックスのB-treeも更新**される。インデックスが3つ（PK + order_id + product_id）あれば、B-treeの更新も3回。しかもB-treeの更新は単純な追記ではなく、ソート順を維持するために適切な位置への挿入・必要に応じたノードの分割が発生する。

#### ディスク消費量

| 状態 | テーブル本体 | インデックス合計 | 全体 |
|------|---:|---:|---:|
| PKのみ | 149 MB | 64 MB | **214 MB** |
| PK + 2インデックス | 149 MB | 126 MB | **276 MB** |

使われないインデックスが**62MB**のディスクを消費している。テーブル本体の約40%に相当する。本番環境で数十億行規模なら数百GBになりうる。

#### 「貼っておけば損はない」に対する説明

「インデックスはSELECTを速くするが、INSERT/UPDATE/DELETEを遅くする。Q1のように使われないインデックスは、読み取りの恩恵ゼロで書き込みコストだけ払っている状態。うちのorder_itemsは10万件INSERTで60%遅くなった。さらにディスクも62MB余分に食っている。インデックスは必要な列にだけ貼るもの」

### 「書籍を読んだだけでは分からなかったこと」

1. **「SELECTが遅くならない」≠「損がない」**: Q1でインデックスの有無によるSELECT速度の差はなかった。オプティマイザは賢くインデックスを無視してくれる。しかし書き込み側では10万件INSERTが60%遅くなっていた。「貼って損はない」の「損」はSELECTではなく書き込みとディスクにかかる
2. **非正規化は「読む行数を減らす設計」**: 集計サマリーテーブルのように目的に合った非正規化でなければ効果がない
3. **選択率の「体感」**: 0.003%→70倍、8%→25%改善、2%→6%改善という具体的な数字を見ることで、「選択率が高いとインデックスが効く」という説明の解像度が上がった
